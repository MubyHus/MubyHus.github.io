相关术语

UP（Uni-Processor）：单核CPU系统，一个时刻只有一个进程在执行，这里只会发生中断和进程之间的抢占关系。

SMP（Symmetric Multi-Processors）：系统有多个处理器单元。各个处理器之间共享总线，内存等等。此时进程/中断都可能会访问资源竞争。

临界区：可以认为就是一个时间内只能有一个执行者的一段代码。

PE (processing element)：可以简单的任务是一个cpu core。

抢占：高优先级任务可运行时，会剥夺低优先级任务的执行，即使此时低优先级任务的时间片还没有用完。

原子操作：不会被打断的操作，该操作如果执行，肯定是执行完成，执行过程中不会受到其他任何干扰。

接下的内容会以arm64 下的SMP 为例，详细介绍一下spin_lock 的实现。这个过程中会涉及一些汇编的内容，在遇到相关代码时再解释。

在涉及到锁一类机制的理解时，最好在脑海中形成这样一个概念：C代码中的任何一个语句都不是原子的，也就是说，任何一句代码都可能被CPU打断，从而切换到其他的代码执行。

形成了这样的概念后，有利于在实际代码中小心对待锁的使用，同时可以更好的理解硬件提供的一系列方式，特别某些执行来保证原子操作。

spinlock的实现

spinlock是内核中最基本的锁机制，其他的锁机制基本上都是再它的基础上衍生出来的。之所以需要锁这样的机制，实际上是说某一段代码，一次只能有一个执行者在跑这段代码，一般来说这样的一段代码，可能是操作硬件资源(读某些硬件寄存器)或者是操作某个内存中的数据。

由于此文章主要关注于spinlock，所以接下说的锁都是默认指的spinlock。spinlock 的特点就是，如果当前操作者获取不到这个锁，就会进行自旋(spin)操作，也就说或一直尝试，直到最终成功获取到。由于这个特性，一般来说希望使用spinlock 的场合，临界区都能尽快执行完。

在接下来的内容中，将会通过对代码的分析，着重解释清楚如下的两个问题：

  1. spinlock 在arm64 上实现的原理；
  2. spinlock 涉及的书籍中，都会特意提到，在由spinlock 保护的临界区的代码中，不能有睡眠的操作。这又是为什么？

//kernel/arch/arm64/include/asm/spinlock.h
static inline void arch_spin_lock(arch_spinlock_t *lock)
{
	unsigned int tmp;
	arch_spinlock_t lockval, newval;

	asm volatile(
	/* Atomically increment the next ticket. */
	/* LL/SC */
"	prfm	pstl1strm, %3\n" //将lock放到L1 cache中，加快访问速度
"1:	ldaxr	%w0, %3\n" //从lock处load 32bit数据放到w0中；
"	add	%w1, %w0, %w5\n" //w0 + 1 << TICKET_SHIFT，就结果放到w1中。
						//注意arch_spinlock_t是4字节对齐的，此时就是对next++(注意字节序)
"	stxr	%w2, %w1, %3\n" //将w1的数据存回lock对应的内存中，操作成功与否的指示保存在w2中
"	cbnz	%w2, 1b\n", //如果w2不为0，则跳转到标号1继续执行；如果w2为0，表示存回内存的操作成功，继续执行
	//....
	)

	/* Did we get the lock? */
"	eor	%w1, %w0, %w0, ror #16\n" //将w0和w0循环右移的数据进行或运算，结果存入w1中
"	cbz	%w1, 3f\n" //如果w1为0，则说明next和owner相等，跳转到标号3执行，此时说明获取到了锁；
					//如果w1不为0，说明没有获取到锁
	/*
	 * No: spin on the owner. Send a local event to avoid missing an
	 * unlock before the exclusive load.
	 */
"	sevl\n"
"2:	wfe\n" //没有获取到锁，CPU进入低功耗模式
"	ldaxrh	%w2, %4\n" //将owner值放入w2中，注意，owner是16bit的
"	eor	%w1, %w2, %w0, lsr #16\n" //w0的next和w2中的owner进行或运算，结果存入w1
"	cbnz	%w1, 2b\n" //如果w1不为0，说明没有获取到锁，继续低功耗等待；如果为0，说明获取到锁
	/* We got the lock. Critical section starts here. */
"3:"//memory barrier操作
	: "=&r" (lockval), "=&r" (newval), "=&r" (tmp), "+Q" (*lock)
	: "Q" (lock->owner), "I" (1 << TICKET_SHIFT)
	: "memory");
}

每个操作数前面都会有一些限定字符，其大致意义如下：

r:表示变量会被先读到通用寄存器，之后的操作直接使用通用寄存器的数据；
m 操作数据是放在内存，而不是通用寄存器中的；
I 表示立即数；
= 表示只写，+ 表示可读写，& 只能作为输出；
Q 表示只使用一个寄存器保存的内存地址，不是offset；

注意一点：所有的输出操作数和输入操作数，按照顺序，可以使用对应的%0、%1、.... 这样的方式访问，同时它们也被按照顺序存入w0 为起始的寄存器中。 所以w0 指代lockval 变量的值，w5 指代(1 << TICKET_SHIFT)。当然在实际汇编出来的代码中，可能变量不一定真的就存放在对应的寄存器，而是由编译器根据实际情况分配，但是这里的汇编代码可以按照这样的逻辑使用这些寄存器。

另外前文中的汇编代码中，使用了一些arm64 对应的汇编指令，这些汇编指令由硬件提供了独占的行为。例如：

1
2
3
4

ldaxr: 该指令原子地从内存中获取数据，对应的物理内存被标记为独占，这个标志会被独占写入指令检查
stxr：该指令检查物理内存的独占标记，如果是本身这个PE之前独占获取，那么此时可以独占写入；
		0表示独占写入成功，1表示失败，数据未写入
ldaxrh：该指令与ldaxr相同，只是读取16bit，其他高位bit被全部设置成0

所以spinlock 这样的锁，并不能单纯靠软件实现，而是架构硬件需要提供这样独占访问的能力。现在我们模拟一下函数被调用的情况，看看实际过程中流程是怎么样。

假设现在有两个执行体，分别在CPU0和CPU1上执行代码，最终都需要获取一个相同的spinlock。此时：

  1. CPU0执行到标号1的代码，将lock对应物理内存值通过ldaxr 存到了w0中，注意此时物理地址被置上了一个标志，只是CPU1才能往该内存写入数据；
  2. 此时CPU1执行到标号1的代码，也将对应物理内存通过ldaxr 存到了它的w0中，执行后续操作；
  3. CPU0修改数据后，通过stxr 指令将修改数据存回物理地址，由于物理地址的独占标志检测通过，数据写入成功；此时CPU1修改了数据，也想写入，但是由于物理内存数据已经被CPU0改写，CPU1只能重新执行标号1，重新读取并改写；
  4. 通过ldxar 和 stxr，从硬件上保证了lock对应的物理内存一次只能有一个执行体去修改。就算执行add 这样的指令时，由于调度流程，CPU0被切走执行其他指令(一般不会，由于关闭了抢占)，导致CPU1先写入成功了，但是CPU0在切回后，依然可以保证CPU0会重新获取内存中数据(被更新后的)，而不会使用旧的数据。
  5. 接下来就是比较lock中的owner和next是否相同。在arch_spin_lock中，next++；在arch_spin_unlock中，owner++；只有相同才代表获取到锁；

为什么不能睡眠？

前文的部分内容详细说明了spinlock 的实现原理，从其中可以看出，自旋锁的正确操作依赖于物理架构提供的一些特殊指令。在arm 上就是一些独占操作指令。这里还有文章最开始的一个问题没有解决，为什么在spinlock 中不能休眠？这个实际上和上锁时会关闭抢占有关。

从void spin_lock(spinlock_t *lock) 函数调用的流程看(调用流程图见前文)，在走到__raw_spin_lock 函数时，第一个作用就是把抢占给关了。这里一个最基本的问题就是，为什么需要关抢占，不关抢占行不行？

要回答这个问题，可以还是实际看不关抢占的流程会是怎么样的：

1
2
3
4

1. CPU0的A进程，获取到锁，正在执行临界区代码；
2. CPU0上可能由于某些资源已经准备好，高优先级任务对应的进程B抢占A进程。
3. B进程中也来获取锁，但是此时owner和next不等，进入WFE；
4. CPU0由于WFE，A进程无法再执行，导致死锁产生；

所以这里必须把抢占关了，这样就可以解决上述问题。而关闭抢占，就会导致另外一个限制，就是临界区代码不能睡眠。同样看看如果睡眠会产生什么问题：

1
2
3

1. CPU0的A进程，获取到锁，正在执行临界区代码，此时调用了睡眠相关api；
2. 睡眠相关api会主动调用schedule，导致CPU调度其他进程，此时B进程被调度；
3. B进程也获取锁，此时会失败，进入WFE；从而导致无法再有其他进程被调度；

上面的讨论都是基于arm 的代码，也就是获取不到锁就WFE。如果没有WFE 呢？同样的道理，B进程会一直占用CPU执行next和owner的对比。由于此时抢占已经被禁止，进程A永远不会被调用到，从而B进程一直无法获得锁。

以上就是在spinlock 中不能进行睡眠的原因。从这些流程里面可以看到，在驱动中使用锁时需要特别注意，需要了解所使用的这类所有什么限制，从而在临界区代码中做一定的规避。

总结

spinlock 的代码，主要的难点还是在于汇编语言的实现部分。这一部分与体系结构强相关，同时又涉及到不同平台的汇编语言的差异。像这里以arm 平台为例，理解这些汇编代码最好的方式就是去arm 官网，下载该体系结构下的汇编文档。这部分的嵌入式汇编代码量并不大，只要可以根据文档，仔细理解每一句汇编的作用，结合硬件架构的能力，就能看到软件代码是如何与硬件结合完成对应功能的。
