系统调用与字符设备驱动

 2020.4.14  2020.5.7

在之前的文章中，已经介绍过字符设备驱动框架，可以参考字符设备驱动框架。在之前讲解设备驱动框架时，只是给出了框架最终建立起来形成的数据结构关系，还有一个比较重要的部分，就是上层对具体驱动设备结点的读写操作，是如何从用户空间到内核空间，并最终调用到具体驱动的fops的。

本文旨在从使用者的角度，厘清驱动完成的fops是最终如何为用户空间提供服务的。这里会初步涉及到文件系统、系统调用等，文档相对也会比较长，为了能更容易理解这部分的内容，这里将会根据开机时的驱动加载顺序，看看一些数据结构和回调都是在什么时候设置好的，最后再从用户层跟踪下来，看看一个常用的open操作是如何走到驱动中的。

系统初始化步骤

系统初始化时，会加载devtmpfs文件系统，所有的设备节点都是在这个文件系统中建立的。整个初始化的流程如下(点击查看大图)： 经过如上图的初始化函数调用流程后，系统中存在一个名为kdevtmpfs 的内核线程。这个内核线程是专门用于建立设备节点，当然如果设备节点的路径没有建立起来，也会同时建立其对应的目录。

设备节点的建立，通常是在设备驱动注册时，即调用到设备驱动模型中的device_add 函数，内部会调用devtmpfs_create_node 这个函数，它实际上就是初始化了一个struct req 变量，然后将其挂到全局的request 链表上，通过wake_up_process 唤醒kdevtmpfs 线程，简略代码如下：

int devtmpfs_create_node(struct device *dev)
{
	const char *tmp = NULL;
	struct req req;

	if (!thread)
		return 0;

	req.mode = 0;
    //....
    //mode默认是0600
	if (req.mode == 0)
		req.mode = 0600;
    //记录设备是块设备还是字符设备
	if (is_blockdev(dev))
		req.mode |= S_IFBLK;

	requests = &req;
	spin_unlock(&req_lock);

    //唤醒内核线程，创建设备节点
	wake_up_process(thread);
    //.....
}

static int devtmpfsd(void *p)
{
    *err = sys_mount("devtmpfs", "/", "devtmpfs", MS_SILENT, options);

	while (1) {
		spin_lock(&req_lock);
		while (requests) {
			struct req *req = requests;

			while (req) {//对所有的req，调用hanlde函数

}

也就说当我们想生成一个设备节点时，实际上是查找设备节点所在的上级目录，然后找到上级目录对应的inode，通过其内部的回调函数mknod 完成的。接下来，就需要确认一下，在devtmpfs 文件系统中的目录，这些目录对应的inode->i_ops 被设置成了什么，其mknod 对应的哪个具体函数。

/dev路径的inode

devtmpfs文件系统注册到系统，是在初始化的devtmpfs_init 这个函数里，这个函数会调用register_filesystem 将devtmpfs 注册进系统。devtmpfs 文件系统对应的struct file_system_type 定义是：

	.name = "devtmpfs",
	.mount = dev_mount, //这个就是文件系统被mount时会被系统回调的函数
	.kill_sb = kill_litter_super,
};

注意这里的dev_mount 这个回调，这就是文件系统被mount 时会被系统调用的。在内核线程kdevtmpfs 对应的线程函数devtmpfsd 中，会有如下的一行语句：

1

sys_mount("devtmpfs", "/", "devtmpfs", MS_SILENT, options);

这里调用的sys_mount 实际就是系统调用的mount 操作，那么它就会回调到dev_fs_type 中的dev_mount这个回调函数。首先看一下系统的mount 是怎么调用到这个回调函数的，mount 系统调用对应的原型在kernel/fs/namespace.c 中：

				     umode_t mode, dev_t dev, unsigned long flags)
{
	struct inode *inode;

	inode = new_inode(sb);

            inode->i_op = &shmem_special_inode_operations;
			init_special_inode(inode, mode, dev);
			break;
		case S_IFREG:
            //....
			break;
		case S_IFDIR: //如果是建立一个文件夹，走这个流程
            //.....
			inode->i_op = &shmem_dir_inode_operations;
			//...
			break;
		case S_IFLNK:
            //.....
			break;
		}
	}
    //....
}

所以，当需要在/dev 下创建设备节点时，会先找到dev 目录的inode，然后调用inode->i_op->mknod，根据上面函数的逻辑，目录对应的inode->i_op 设置成了shmem_dir_inode_operations。

所以inode->i_op->mknod函数实际就调用到了shmem_mknod ，路径在kernel/mm/shmem.c 。

实际上，shmem_mknod 也会调用上面提到的shmem_get_inode ，但是此时传入的mode 是在devtmpfs_create_node 赋值的0600|S_IFCHR ，所以走的是shmem_get_inode 函数的default流程，内部会调用init_special_inode 函数，指定inode 的i_fop ：

void init_special_inode(struct inode *inode, umode_t mode, dev_t rdev)
{
	inode->i_mode = mode;
	if (S_ISCHR(mode)) {
		inode->i_fop = &def_chr_fops; //指定i_fops为def_chr_fops
		inode->i_rdev = rdev;//i_rdev保存了设备号
	}
    //....
}

至此，我们已经知道了生成的设备节点是如何被指定了默认的操作函数集的。总结一下：

  1. 系统初始化时，挂载了devtmpfs 到/dev，这个目录对应inode->i_op 设置为shmem_dir_inode_operations;
  2. 当在/dev 下创建设备节点时，会调用到其路径的上层目录对应inode，使用上层目录的操作函数中的mkmod，即shmem_dir_inode_operations->mkmod;
  3. 通过shmem_dir_inode_operations->mkmod，最后创建了设备节点，并将设备节点对应的inode->i_fop 设置为def_chr_fops;

const struct file_operations def_chr_fops = {
	.open = chrdev_open,
	.llseek = noop_llseek,
};

可以看到上层用户对设备节点的open 操作，最后是获取到设备节点对应inode 的i_fop，而这个恰恰是设备节点生成时被设置为def_chr_fops 的。

这样上层用户的open 最后执行的就是def_chr_fops 指定的chrdev_open 函数。在chrdev_open 这个函数里面，实际就是根据inode->i_rdev 得到设备节点对应的设备号，之后从字符设备框架的cdev_map 中最终得到对应的struct cdev。据此获取到struct cdev 中的ops，这个ops 就是字符设备注册时由具体的设备驱动指定的。并且会将flip 中的f_op 替换成具体驱动的ops，然后调用到设备驱动的open 函数：

static int chrdev_open(struct inode *inode, struct file *filp)
{
	const struct file_operations *fops;
	struct cdev *p;
    struct cdev *new;

	spin_lock(&cdev_lock);
	p = inode->i_cdev;
	if (!p) {
        //....
		kobj = kobj_lookup(cdev_map, inode->i_rdev, &idx);
        //通过inode->i_rdev设备找到了cdev
		new = container_of(kobj, struct cdev, kobj);
		p = inode->i_cdev;
        //....
	}
    //....
    //获取cdev->ops
	fops = fops_get(p->ops);
	if (!fops)
		goto out_cdev_put;

    //替换了filp->f_op为驱动的fops
	replace_fops(filp, fops);
	if (filp->f_op->open) {
		ret = filp->f_op->open(inode, filp);
		if (ret)
			goto out_cdev_put;
	}
    //....
}

也就是说，经过open 操作，对应的struct file 变量内部的f_op 已经被替换成对应的设备驱动指定的file_operations，之后的read/write 操作，就可以调用到设备驱动中的实现。

read/write 这些系统调用定义在kernel/fs/read_write.c 中，其内部最后调用到的就是file->f_op->read/write ，由于此时file 的操作函数集已经被替换成具体设备驱动的，所以调用的就是驱动的函数。

这样从用户层到驱动的操作就通过这个比较复杂的方式建立好了关联，驱动中实现的操作函数，也通过这个方式和设备节点绑定起来，最终通过系统调用实现与设备的数据交换。

总结

从整个用户调用到最终和设备驱动的操作绑定，实际上都是通过各种回调实现的。整个过程最重要的就是需要知道inode 的各种操作函数集合是在什么时候被设置的，设置成了什么。抓住了这个重点，就能很好理解最终调用到驱动的过程。
