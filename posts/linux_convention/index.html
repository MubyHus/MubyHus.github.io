#define THIS_MODULE(&__this_module)

__this_module 是由内核模块的工具链为当前模块产生的一个struct module 对象。

struct file_operations 这个结构体中会有指向struct module 对象的指针，主要是为了保证file_operations 中的函数在被调用时，模块不会被从系统中卸载。如果驱动不是以模块的方式存在，而是直接编译进内核，那么THIS_MODULE 是空指针，没有任何作用——模块编译进内核，加载是在开机时，整个过程不会卸载。

2.

Linux驱动中，用的比较多的是module_init，这里涉及内核提供的initcall 相关机制。

关于该机制，可以参考Linux Insides的这个部分。这里结合实际可能遇到的代码，再补充几个需要注意的地方。

首先，所有开机加载的驱动(非模块加载)，其init 函数在编译后可以在System.map 中找到，一般如果需要确认驱动的加载顺序，可以在该文件中查找。所有的init 函数根据其实际的initcall 顺序排列，并且是没有模块卸载函数的。

其次，如果驱动是模块形式加载的，那么无论是用哪种xxx_initcall 实际最后都是用的都是module_init，同时模块的卸载函数也是有用的，在rmmod 时会被调用到。

#define module_exit(x)	__exitcall(x);

#else /* 表示是动态加载的模式 */

#define early_initcall(fn)		module_init(fn)

//在模块代码中定义了init_module和cleanup_module函数，分别是对应传进来函数的别名
//这两个函数insmod和rmmod时需要使用的
#define module_init(initfn)					\
	static inline initcall_t __maybe_unused __inittest(void)		\
	{ return initfn; }					\
	int init_module(void) __attribute__((alias(#initfn)));

#define module_exit(exitfn)					\
	static inline exitcall_t __maybe_unused __exittest(void)		\
	{ return exitfn; }					\
	void cleanup_module(void) __attribute__((alias(#exitfn)));

3

在实际工作中，可能会遇到某些设备驱动之间有加载的依赖关系要求。有两种方法可以解决这个问题：

  • 最容易想到的是通过不同的init 函数，调整驱动的initcall 的位置。不过这里有个弊端，因为一般写的可能都是驱动程序，大部分用的都是module_init 定义加载函数，比它更晚加载的就是late_initcall 这样的，选择比较少(因为module_init 对应是device_initcall 级别)；
  • component 机制是另外一种内核提供的，可用于顺序初始化的方法，这个机制比较灵活，可以很好的解决驱动加载顺序的问题。使用起来相对简单。
